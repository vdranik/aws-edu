*EBS - Elastic Block Storage*
Может быть прикреплен к любому инстансу, который нах. с ним в одной зоне, а так же прикреплен и откреплен к любому другому
Главный показатель - количество входящий и исходящих операций в секунду (определяют скорость), измеряется в гагабит в сек (IOPS).
Чем больще хранилище, тем больше скорость передачи данных.
Каждый инстанс ЕС2 должен иметь корневой(root) объем, который закреплен за ним (как жесткий диск).
Если при создании ЕС2 не убрать галочку на шаге Storage "Delete on Termination", то при удалении ЕС2 также будет удален корневой EBS!
Лучше эту галочку уобрать и при удалении ЕС2 иснстанса хранилище останется.
В любой момент можно добавить объем.
Snapshot - образ, не самих данных, а настроек. Такой шаблон будет полезен при создании аналогичных EBS.


Security groups
                         
                        NACL
Subnet1 (ЕС2 --- SG) --- | ---
                         |	  |
                         |	 Load balancer --- Internet Gateway ---
                         |	  |	
Subnet2 (ЕС2 --- SG) --- | ---

На этой схеме важно, чтобы SG по правилах совпадали, иначе возможны разного рода ошибки.

SG чем-то похожа на NACL, работает как фаерволл, блокирует или разрешает трафик.
НО в отличие NACL, который работает на уровне подсетей, Security groups работают на уровне ЕС2 инстанса!


Кроме этого, механизм правил контроля отличнается:
 - у NACL есть пронумерованные правила (номер определет очередность вступления в силу) + тут можно создавать правила блокирующие и разрещающие трафик. Работает на уровне Subnets.
 - у Security groups все правила ступают в силу и можно создавать только разрещающие правила (все остальное будет заблокировано).
 Не имеет номеров для правил. Работает на уровне ЕС2.

При запуске инстанса можно прикрепить одну или более групп безопасности.
В SG добавляются разрешающие правила для привязаных к ним инстансов. Эти правила можно редактировать в любой момент и они сразу вступают в силу, как только она была прикреплена к инстансу. Если SG несколько, абсолютно все правила будут работать для трафика.

Новосозданная вами SG не будет иметь правил, а занчит запрещает весб трафик.
SG default котор. создается AWS имеет базовое разрещающее правило.
SG имеет inbound/outbound rules. Любое создание правило будет разрешать трафик.

SG автоматически разрешают выйти трафику который попал к ним извне! 
Т.е. тот трафик который прошел SG inbound rules не будет проходить outbound rules чтобы покинуть инстанс.

Важно, чтобы правила в NACL не противоречили правилам в SG.
